Mission-profile chain strip — mockups

Migration step 4.2: cheap, clickable mockups of the one UI element with no precedent in the existing tools — the visible sequence of mission stages you add to, reorder, and swap. Three layout variants below, all driven by one shared fake mission (the worked example from ARCHITECTURE.md), so a change made in any variant appears in all three. No real physics, no core imports — every number is canned.

Try: click a stage to select it · drag a stage to reorder · hover a connector for the + insert point · ⇄ swaps the module · ✕ removes · the failure toggle below shows the diagnostic + blocked-downstream states.

A — horizontal card strip

Full cards across the top, under the date bar. The mission reads left-to-right like a timeline and each stage carries its own key numbers, so the whole profile is glanceable without touching the panel. Costs the most vertical space of the three; long missions scroll sideways.

C — compact “metro line” strip

What these mockups are meant to settle

  • Orientation and placement — top strip (A, C) vs. sidebar column (B), and how much space the chain deserves relative to the 3D views.
  • Stage density — should a stage show its numbers in the strip (A, B) or only in the panel (C)?
  • Packet-type pills — the little “ship-state” tags on the connectors: informative or noise?
  • How failure reads — toggle the failure and judge: is “diagnostic here, blocked-with-params-kept downstream” obvious in each variant?
  • Where actions live — add on connectors, swap/remove on cards (A, B) vs. everything in the panel card (C). The panel card always carries them regardless.
  • Reordering feel — drag a stage in each variant; which is least fiddly?

None of this code is meant to survive: the real strip will be rebuilt against MissionPlanner/core in step 4.3 once a direction is picked. Mix-and-match verdicts are welcome (e.g. “B’s cards, but across the top” or “C, but with Δv on the labels”).
